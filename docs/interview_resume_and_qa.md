### 标准版

```text
项目名称：
掌柜智库 RAG 知识库问答系统

使用技术：
Python、FastAPI、LangGraph、LangChain、OpenAI 兼容大模型、BGE-M3、FlagEmbedding Reranker、Milvus、MongoDB、MinIO、SSE、MinerU、MCP Web Search、Loguru

项目描述：
该项目是一个面向设备/产品说明书场景的 RAG 知识库问答系统，支持 PDF/Markdown 文档导入、知识切片构建、向量化入库以及基于多路召回和大模型生成的智能问答。系统整体分为“导入建库”和“查询问答”两条核心链路，形成从文档解析到答案生成的完整闭环。

项目内容：
1. 基于 LangGraph 设计并编排文档导入流程，将文件上传、PDF 转 Markdown、图片处理、文档切块、主体识别、向量生成、Milvus 入库等步骤组织为可追踪工作流。
2. 实现 Markdown 图片增强处理，结合图片上下文调用视觉模型生成摘要，并上传至 MinIO，提升图文内容在知识库中的可检索性。
3. 设计文档切块策略，结合标题语义切分、超长内容二次切分与短内容合并，提升知识切片质量和召回效果。
4. 实现基于 LLM 的文档主体识别能力，抽取 item_name 并建立独立向量索引，在查询阶段先确认主体再做定向召回，降低跨文档误召回。
5. 基于 BGE-M3 构建稠密向量与稀疏向量混合检索方案，并使用 Milvus 完成 HNSW + 稀疏倒排索引的向量检索。
6. 在查询链路中引入问题改写、HyDE 假设答案检索、多路召回、RRF 融合和 Rerank 精排，提升复杂问题的召回率和答案相关性。
7. 基于 FastAPI + SSE 实现流式问答输出，支持前端实时接收任务进度、模型增量内容和最终答案，优化交互体验。
8. 使用 MongoDB 存储历史会话，支持多轮问答上下文拼接、历史查询与删除；使用 MinIO 托管图片资源，支持图文混合问答返回。
9. 抽离配置、日志、Prompt、任务状态和外部服务客户端，形成相对清晰的工程分层，提升系统可维护性与扩展性。
```

## 二、模拟面试题与参考回答

### 1. 这个项目的核心目标是什么？

参考回答：
该项目的核心目标是把设备说明书、产品手册等文档导入知识库，并在查询阶段结合主体识别、混合检索、多路召回、重排序和大模型生成，为用户提供更准确的问答服务。它本质上是一个面向垂直文档场景的 RAG 系统。

### 2. 项目整体架构是怎样划分的？

参考回答：
整体可以拆成六层：接口层、LangGraph 流程层、模型能力层、存储与外部服务层、基础设施层和前端页面层。接口层主要由 FastAPI 提供上传、查询和流式输出能力；流程层由导入图和查询图组成；模型层负责 LLM、Embedding 和 Reranker；存储层主要是 Milvus、MongoDB 和 MinIO。

### 3. 为什么这里使用 LangGraph？

参考回答：
因为项目不是一个单步骤请求，而是一个多节点、带分支、可并行的流程。比如导入流程要根据 PDF 或 Markdown 走不同路径，查询流程里又要并行执行普通检索、HyDE 检索和联网搜索。用 LangGraph 可以把流程节点化、状态化，便于扩展、排查和维护。

### 4. 导入链路的完整流程是什么？

参考回答：
导入链路从文件上传开始，文件先落盘到本地输出目录。如果是 PDF，会先调用 MinerU 转 Markdown；之后统一进入图片处理节点，对图片进行上下文提取和摘要增强；接着做文档切块、主体识别、BGE-M3 向量生成，最后将 chunk 数据和主体向量写入 Milvus。

### 5. 文档切块为什么这么重要？

参考回答：
因为切块质量直接影响召回质量。如果切得过大，召回噪音会多；切得过碎，语义会被打断。这个项目在切块时结合了 Markdown 标题语义切分、长文本二次切分和短文本合并，目的是在“语义完整”和“检索粒度”之间做平衡。

### 6. 为什么要做图片摘要增强？

参考回答：
很多设备说明书的关键信息都在图片里，比如按键说明、接线图、结构示意图。如果只检索文字，很多重要信息会缺失。项目会扫描 Markdown 中的图片，结合图片前后文调用视觉模型生成摘要，并把图片上传到 MinIO，让图片信息变成可检索、可展示的文本资源。

### 7. 什么是 `item_name`，为什么它很关键？

参考回答：
`item_name` 可以理解为文档的主体名称，比如某个设备型号、产品名称。它很关键，因为设备手册里经常会有重复章节，例如“开机方法”“故障排查”。如果不先限定主体，系统可能把别的设备内容召回出来。项目通过先识别文档主体并建立单独向量库，在查询前先确认主体，再在指定主体范围内检索，从而减少误召回。

### 8. 为什么要使用 BGE-M3 的混合检索？

参考回答：
BGE-M3 同时支持稠密向量和稀疏向量。稠密向量更偏语义理解，稀疏向量更保留关键词和术语匹配能力。对于设备型号、功能名、菜单项这类词，单纯稠密检索可能会丢失精确信息，所以项目采用稠密加稀疏的混合检索来兼顾语义和关键词。

### 9. HyDE 在这个项目里解决了什么问题？

参考回答：
HyDE 的作用是提高召回率。它先让模型根据用户问题生成一个假设性的答案，然后用“问题 + 假设性答案”去做向量检索。这样可以补足用户原问题里没有表达完整的语义信息，尤其适合用户提问很短、上下文不足的情况。

### 10. RRF 和 Rerank 有什么区别？

参考回答：
RRF 主要用于多路召回结果融合，比如普通向量检索结果和 HyDE 检索结果怎么合并；Rerank 则是在候选结果已经出来之后，再使用更精确的交叉编码模型对问题和文档进行相关性打分，完成最终精排。可以理解为 RRF 是融合阶段，Rerank 是精排阶段。

### 11. 为什么查询链路里还要接入联网搜索？

参考回答：
因为本地知识库不一定覆盖所有信息，尤其是一些补充性背景知识、公开网页资料或者知识库没有及时更新的场景。联网搜索可以作为补充召回源，提高系统在边界问题上的回答完整性。

### 12. 项目里的 SSE 解决了什么问题？

参考回答：
SSE 主要解决大模型问答的实时交互问题。查询走流式模式时，系统先返回 `session_id`，后台执行 LangGraph 查询流程，前端通过 SSE 长连接持续接收节点进度、模型增量 token 和最终答案。这样用户不用一直等待整段回答生成完，体验会更好。

### 13. MongoDB 在这里的主要作用是什么？

参考回答：
MongoDB 主要用于存储多轮会话历史，包括用户问题、助手回答、重写问题、主体名称和时间戳等信息。系统会把历史记录拼接进 Prompt 中，让模型具备多轮上下文理解能力，同时也支持历史查询和删除。

### 14. MinIO 的使用场景是什么？

参考回答：
MinIO 主要用来托管图片资源。项目在处理图片增强时，会把图片上传到 MinIO 并生成可访问地址，用于 Markdown 内容回写、检索结果回显以及最终答案附图展示。

### 15. 你觉得这个项目最有价值的设计点是什么？

参考回答：
我觉得有三个点最有价值。第一是用 LangGraph 把复杂流程拆成清晰节点，提升了可维护性。第二是通过主体识别 + 混合检索 + HyDE + RRF + Rerank 形成了完整的检索增强链路。第三是把图片信息纳入知识库，解决了传统文本 RAG 对图像内容覆盖不足的问题。

### 16. 如果让你继续优化这个项目，你会从哪些方向入手？

参考回答：
我会从三方面继续优化。第一，任务状态目前基于内存字典，只适合单进程，我会改造成 Redis 或数据库持久化。第二，建立离线评测集，对召回率、命中率和答案质量做量化评估。第三，补齐 README、环境变量说明、部署文档和测试覆盖，提高工程成熟度。

### 17. 你在这个项目中的亮点可以怎么讲？

参考回答：
我可以重点讲三点。第一，我对项目做了流程拆分和架构梳理，明确了导入链路和查询链路。第二，我理解并串联了主体识别、混合检索、HyDE、RRF 和 Rerank 这套 RAG 增强方案。第三，我能够从工程视角看到它的可扩展点和不足，比如流式输出、图片增强、任务状态持久化和文档化建设。

## 三、八股化速答版

### 适合 30 秒左右回答的问题

#### 1. 请你简单介绍一下这个项目。

速答：
这是一个面向设备说明书场景的 RAG 知识库问答系统，分成导入建库和查询问答两条链路。导入侧完成 PDF/Markdown 解析、图片增强、切块、主体识别和向量入库；查询侧完成主体确认、多路召回、RRF 融合、Rerank 精排和大模型答案生成。

#### 2. 项目为什么要分导入链路和查询链路？

速答：
因为这两个链路职责完全不同。导入链路负责把原始文档变成可检索知识；查询链路负责把用户问题转成高质量答案。分开后结构更清晰，也更方便维护和扩展。

#### 3. 为什么项目里要先做主体识别？

速答：
因为设备手册之间有很多相似章节，不先确认主体会导致跨文档误召回。先识别 `item_name`，再在对应主体范围内检索，能提高答案准确率。

#### 4. HyDE 的核心思想是什么？

速答：
先让模型根据用户问题生成一段假设性答案，再用“问题 + 假设答案”去做检索，补充语义信息，提高召回率。

#### 5. RRF 和 Rerank 的区别是什么？

速答：
RRF 是多路召回结果融合，Rerank 是对候选文档做精排打分。前者解决“怎么合”，后者解决“怎么排”。

#### 6. 为什么这里要做流式输出？

速答：
因为大模型生成答案需要时间，流式输出可以让前端实时看到进度和增量内容，用户体验更好。

#### 7. MongoDB 和 Milvus 分别负责什么？

速答：
Milvus 负责存储向量和做检索，MongoDB 负责存储多轮会话历史。

#### 8. 为什么引入 MinIO？

速答：
因为项目里有大量图片资源，需要一个对象存储系统统一托管，方便图片回写、展示和图文问答返回。

#### 9. 项目的工程不足是什么？

速答：
目前任务状态还是内存级管理，只适合单机单进程；README 和部署文档也不够完善；部分节点里还有实验性代码，后续需要收敛。

#### 10. 你从这个项目里学到什么？

速答：
我学到如何从工程角度搭建一个完整的 RAG 系统，不只是调用大模型，而是把文档处理、混合检索、排序优化、流式输出和多轮会话整合成一条完整链路。
